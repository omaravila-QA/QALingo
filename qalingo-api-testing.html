Concept 01
What is an API?
The contract between frontend and backend — and where most production bugs live.
Definition
An API (Application Programming Interface) is a contract between two systems. The client sends a request following the contract's rules. The server responds with data or an action result. REST APIs use HTTP — the same protocol as web browsers. As QA, you test that contract holds under all conditions.
The Request-Response Cycle
Every API interaction has the same structure:

Request: Method + URL + Headers + Body (optional)
Response: Status Code + Headers + Body (JSON/XML)

Example: POST /api/users with JSON body → returns 201 Created + new user object
HTTP Request anatomy
POST /api/users ← Method + Endpoint Host: api.qalingo.com Content-Type: application/json ← Headers Authorization: Bearer eyJhbGciOiJIUzI1Ni... { ← Request Body "name": "Watson", "email": "watson@qalingo.com", "plan": "pro" } ─── Response ───────────────────────────── 201 Created Content-Type: application/json { "id": "usr_abc123", "name": "Watson", "email": "watson@qalingo.com", "status": "pending", ← QA: is this correct? "created_at": "2024-01-20T10:30:00Z" }
💡 Why QA needs API testing
UI tests can only verify what the user sees. API tests verify what actually happens in the system — the data that gets saved, the logic that runs, the security that holds. 80% of production bugs happen in the backend. You need API testing to find them.

Concept 02
HTTP Methods
Each method is a verb. Know what it does — and how to break it.
GET Read data — no changes
POST Create new resource
PUT Replace entire resource
PATCH Update partial fields
DELETE Remove the resource
QA Testing Matrix per Method
GET: Test with valid ID, invalid ID, unauthorized user, non-existent resource
POST: Valid body, missing required fields, duplicate data, invalid data types
PUT: Full replacement — test that ALL fields update, not just changed ones
PATCH: Partial update — test that ONLY specified fields change
DELETE: Successful delete, delete non-existent, delete without permission, double delete

Concept 03
HTTP Status Codes
The server's first word. Know what every code means for QA.
200
OK
Request succeeded. GET, PUT, PATCH responses.
QA: verify body contains correct data
201
Created
Resource created. Expected for POST requests.
QA: verify Location header + new ID in body
204
No Content
Success with no response body. Common for DELETE.
QA: verify resource is actually gone in DB
400
Bad Request
Client sent malformed or invalid data.
QA: verify error message describes what's wrong
401
Unauthorized
Missing or invalid authentication token.
QA: test with expired/missing/wrong token
403
Forbidden
Authenticated but not authorized for this action.
QA: test role boundaries — can user A access user B's data?
404
Not Found
Resource doesn't exist at this endpoint.
QA: test with non-existent IDs and wrong paths
409
Conflict
Duplicate data — e.g., email already exists.
QA: test duplicate registration, duplicate orders
422
Unprocessable
Syntactically correct but semantically invalid.
QA: send wrong data types, invalid enum values
500
Server Error
Server crashed. Never acceptable in production.
QA: report immediately — 500s are always bugs

Concept 04
Postman Basics for QA
The essential tool — collections, environments, and variables.
Core Postman Concepts
Collection: A folder of related API requests — e.g., "User Management" containing all user endpoints

Environment: A set of variables for different contexts — {{base_url}} = staging vs production

Variable: Reusable values — {{token}}, {{user_id}}, {{base_url}}

Test Script: JavaScript code that runs after a response to verify expectations automatically
Postman — Using Variables
// In Postman request: GET {{base_url}}/api/users/{{user_id}} Authorization: Bearer {{token}} // Environment variables: base_url = https://staging.qalingo.com user_id = 42 token = eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9... // Set a variable from response (in Tests tab): const response = pm.response.json(); pm.environment.set("user_id", response.id); pm.environment.set("token", response.access_token);

Concept 05
Authentication Testing
The security layer — where QA prevents data breaches.
Auth types QA must test
API Key: x-api-key: abc123 — test with invalid, expired, missing key

Bearer Token (JWT): Authorization: Bearer eyJ... — test expired token, wrong signature, no token

Basic Auth: Authorization: Basic base64(user:pass) — test wrong credentials

OAuth 2.0: Access tokens with scopes — test accessing resources outside your scope
Security Test Cases — Never Skip These
1. No token: Remove Authorization header entirely → expect 401
2. Expired token: Use a JWT with past exp claim → expect 401
3. Wrong token: Use another user's valid token → expect 403
4. Tampered token: Modify the JWT payload → expect 401
5. Privilege escalation: User tries to access admin endpoint → expect 403
6. IDOR: User A requests User B's private data → expect 403

Concept 06
Assertions & Postman Tests
Automate your verification — don't just look at responses, assert them.
pm.test() — Postman's assertion engine
Assertions run automatically after every request. Instead of manually checking each response, Postman's pm.test() evaluates conditions and reports pass/fail — just like unit tests for your API. These go in the Tests tab of every request.
Postman Tests — pm.test() patterns
// 1. Status code assertion pm.test("Status is 201 Created", () => { pm.response.to.have.status(201); }); // 2. Response time pm.test("Response time under 500ms", () => { pm.expect(pm.response.responseTime).to.be.below(500); }); // 3. Body field assertions const body = pm.response.json(); pm.test("User has correct email", () => { pm.expect(body.email).to.equal("watson@qa.com"); }); pm.test("Status is pending on creation", () => { pm.expect(body.status).to.equal("pending"); }); pm.test("ID is present", () => { pm.expect(body.id).to.exist(); }); // 4. Schema validation pm.test("Response has required fields", () => { pm.expect(body).to.have.keys(['id','name','email','status','created_at']); }); // 5. Array length pm.test("Returns at least 1 user", () => { pm.expect(body.users.length).to.be.above(0); });
Live assertion results — what Q sees when running your tests:
PASSpm.test("Status is 201 Created", () => pm.response.to.have.status(201))
PASSpm.test("Response time under 500ms", () => expect(responseTime).below(500))
PASSpm.test("User email matches", () => expect(body.email).equal("watson@qa.com"))
FAILpm.test("Status is pending", () => expect(body.status).equal("pending")) → got "active"
PASSpm.test("ID is present", () => expect(body.id).to.exist())
💡 The one failing assertion
Notice the 4th assertion fails — the user was created with status "active" instead of "pending". That's a real bug — users should require email verification before being active. Without assertions, you'd have to manually read every response. With assertions, QALingo catches it automatically every time.

Concept 07
Error Scenario Testing
The happy path is easy. The bugs live in the errors.
Error Scenarios Every QA Must Test
Missing required fields: Send POST without required body fields → 400
Invalid data types: Send string where integer expected → 422
Invalid enum values: Send plan="superduper" when only free/pro valid → 422
Empty body: Send POST with {} or no body → 400
SQL injection: Send {"email": "' OR 1=1--"} → 400 (never 500)
Oversized payload: Send 10MB JSON body → 413
Timeout: Slow endpoint with 30s response → client timeout

Concept 08
API + SQL Cross-Validation
The most powerful QA technique — verify API claims against the database.
The Complete Validation Flow
Step 1: Call the API — POST /api/users → 201 Created
Step 2: Verify the API response body — correct fields, correct values
Step 3: Query the database — SELECT * FROM users WHERE email = '...'
Step 4: Compare — does the DB reflect exactly what the API claimed?
Step 5: Check side effects — were any other tables affected?
API call + SQL verification
-- Step 1: API call POST /api/users Body: {"name":"Watson","email":"watson@qa.com","plan":"pro"} Response: 201 {"id":"usr_abc","status":"pending",...} -- Step 2: Verify in database (run in DB client) SELECT id, name, email, status, plan, created_at FROM users WHERE email = 'watson@qa.com'; -- Expected result: -- id: usr_abc (matches API response) ✅ -- status: pending (not active) ✅ -- plan: pro ✅ -- created_at: today's date ✅ -- Step 3: Check no duplicate was created SELECT COUNT(*) FROM users WHERE email = 'watson@qa.com'; -- Expected: 1 (not 2)
💡 The API+SQL combo catches what neither alone can
The API says 201 Created. You check the response — looks perfect. But the DB has the status as "active" instead of "pending." Or the created_at timestamp is yesterday. Or there are two records. Only the database cross-check catches these bugs. This technique is what separates a mid-level QA from a senior one.

Top 10 API QA Interview Questions
What is the difference between 401 and 403?SeniorShow →
401 Unauthorized means the request has no valid authentication — the user isn't identified. 403 Forbidden means the user IS authenticated but doesn't have permission for this action. Think: 401 = "Who are you?", 403 = "I know who you are, but no."
What is IDOR and how do you test for it?SecurityShow →
IDOR (Insecure Direct Object Reference) is when a user can access another user's resource by changing an ID in the URL. Test it by: (1) Log in as User A, (2) Get your resource URL e.g. /api/orders/123, (3) Change 123 to another user's ID, (4) If you can see their data — that's an IDOR bug. Always test with two different user accounts.
What would you check in a POST /login response?MidShow →
Check: Status 200 for valid credentials (not 201). Token present in response body or cookie. Token expiry is set correctly. 401 for wrong password. Account lockout after X failed attempts. No sensitive data in error messages (don't reveal if email exists). Response time is consistent regardless of whether email exists (prevent timing attacks).
What is the difference between PUT and PATCH?BasicShow →
PUT replaces the entire resource — if you send PUT with only {name: "Watson"}, the email and plan fields should be cleared/reset. PATCH updates only the fields you send — if you send PATCH with {name: "Watson"}, only the name changes and everything else stays the same. Test this by calling GET after each to verify exactly which fields changed.
How do you test an API without a UI?BasicShow →
Use Postman to send requests manually and inspect responses. Write pm.test() assertions to automate verification. Use Newman to run collections in CI/CD. Read the API documentation or Swagger/OpenAPI spec to understand expected behavior. Use curl for quick command-line testing. Validate responses against the JSON schema.
A POST endpoint returns 200 instead of 201. Is this a bug?MidShow →
It depends on the API contract. Technically: yes — REST convention says POST that creates a resource should return 201 Created. However, if the API documentation explicitly states 200 for this endpoint, it's not a bug — it's a documentation decision. As a QA: raise it as a defect referencing REST standards, and let the team decide. Also verify the resource was actually created regardless of the status code.
What is the difference between functional and non-functional API testing?SeniorShow →
Functional: Does the API do what it should? Correct status codes, correct response body, correct database changes, correct error messages. Non-functional: How well does it perform? Response time under load (performance testing), behavior with 1000 concurrent users (load testing), security vulnerabilities (security testing), behavior with malformed input (robustness testing). Both are required for a complete API test suite.
How do you test an API endpoint that sends emails?MidShow →
Use a test email service like Mailtrap, MailHog, or Mailosaur — they catch emails in a test inbox without sending them. Verify: (1) Email was sent at all, (2) Correct recipient, (3) Correct subject and body content, (4) Links in email work, (5) No email sent on failed requests, (6) No duplicate emails on retry. Also check the database to confirm the email_sent flag or timestamp was recorded.
What do you check when testing pagination?MidShow →
Test: First page returns correct count. Last page has remaining items (might be less than page size). Page beyond last returns empty array or 404, not an error. Sorting is consistent across pages — no item appears twice or is skipped. Total count in response matches actual total. page=0 or page=-1 handled gracefully. Limit=0 — what happens? Very large limit — does the API protect itself?
How do you handle test data in API testing?SeniorShow →
Best practices: Create fresh data at the start of each test (don't rely on existing data). Clean up after tests — delete what you created. Use Postman pre-request scripts to create prerequisite data. Use environment variables to pass IDs between requests. In CI/CD, use a dedicated test environment with a seeded database. Never run destructive tests against production. Use data factories or fixtures for consistent test data.
